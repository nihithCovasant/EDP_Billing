1. Purpose of the Document
This Product Requirements Document (PRD) defines the business, functional, technical, and non-functional requirements for the AI Automationification of the MOFSL Wealth Management EDP Billing process using the Covasant Automation Management Suite (CAMS).
It serves as the authoritative reference for all business stakeholders, technology teams, and delivery partners throughout the project lifecycle providing a single source of truth that ensures alignment, controlled scope, and predictable delivery.
This document covers Automation design for (Pre-Requisite Automation, EDP Automation, Alerting Automation), integration details, data requirements, guardrails, RACI, assumptions, and out-of-scope boundaries for the EDP Billing Automationification initiative.
2. Business Problem Statement
EDP (End-of-Day Processing) covers all MOFSL back-office activities that run after the stock market closes at approximately 16:30 IST. These activities compute trades across seven market segments — Cash, F&O, CD, SLBM, MCX, NCDEX, and MTF based on trades executed across all MOFSL client accounts. MOFSL operates CBOS (Central Back Office System) which orchestrates all EDP processes.
Key business challenges driving this initiative:
Manual file retrieval: Pre-requisite files (from CDSL, NSDL, NSE, BSE, and other exchanges) is being manually downloaded and placed in designated folders before EDP can commence. Any delay or misplacement directly delays the entire EDP cycle.
No automated pre-requisite monitoring: There is no automated Automation to periodically check whether required files have arrived, leading to ad-hoc manual checks, missed thresholds, and late EDP starts.
No automated email escalation: Failures are triaged manually by operators with no structured escalation matrix, causing inconsistent responses and missed notifications.
No immutable audit trail: There is no end-to-end log of file arrival times, segment completion, sub-process outcomes, or human interventions creating audit and compliance exposure.
Automationification via CAMS addresses these by automating pre-requisite file monitoring, EDP segment orchestration, configurable retry logic, structured failure handling, automated alerting, and a full immutable audit trail all within the defined EDP processing window.

3.4 Current As-Is Process
At approximately 15:30 IST, operations staff manually download required pre-requisite files from CDSL, NSDL, NSE, BSE, MCX, NCDEX portals.
Files are manually placed in designated locations within the MOFSL internal network shared folder.
A manual check is performed to confirm all required files are present in the correct folders. There is no threshold logic or automated wait.
If files are missing, staff manually follow up with exchange/depository portals and retry. No systematic retry policy exists.
Once files are confirmed, the EDP process is manually triggered via CBOS for each of the seven segments sequentially.
Each segment runs a checklist of sub-processes via CBOS. Failures are identified manually by monitoring CBOS output.
On completion, staff manually compile a status report and notify relevant teams via ad-hoc email.

6. Automation Initiation Model
CORE ARCHITECTURAL PRINCIPLE — AUTOMATION INITIATION
Timer-driven: The Pre-Requisite for Automation is initiated at a pre-configured time (e.g. 15:30 IST). The Automation begins polling the designated network folder at the scheduled time.
Once the Pre-Requisite Automation confirms all required files are present, it triggers the EDP Automation. The EDP Automation then builds and executes the segment processing DAG per the execution rules below.
8. Proposed Flow (To-Be)
8.1 Redesigned Workflow — Overview
Automated initiation: Pre-Requisite Automation is initiated by Timer driver at ~15:30 IST — no manual trigger required.
Pre-requisite file monitoring: Pre-Requisite Automation polls the MOFSL network shared folder at configured intervals, waiting for all required files up to a configurable threshold time.
Threshold breach handling: If required files are not present by the threshold, the Pre-Requisite Automation triggers the Alerting Automation (failure report to Ops and IT) and stops. EDP does not start.
EDP trigger: Once all files are confirmed, the Pre-Requisite Automation triggers the EDP Automation via CBOS API.
DAG execution — Dependent segments: EDP Automation executes all seven segment checklists sequentially via CBOS API. Each segment is a Dependent step, failure stops the process.
Retry and failure handling: On failure, configured retries are attempted.
Audit logging: Audit Automation writes a structured log entry to CBOS for every step trigger, outcome, retry, and human override.
Alerting: Alerting Automation dispatches structured email reports via MOFSL SMTP on every failure (with error details) and on overall EDP success.

10. Non-Functional Requirements
10.1 Logging & Audit Trail
Every Pre-Requisite Automation poll cycle must be logged: timestamp, files expected, files found, files missing, poll outcome.
Every EDP Automation action must be logged: step/sub-process triggered, CBOS API call details, response status, error code (if any), retry count, duration, timestamp.
Every Alerting Automation dispatch must be logged: alert type, recipient DL, email subject, dispatch timestamp, SMTP delivery status.
All logs must be immutable — no modification or deletion post-write.
Automation activities will be available in CAMS control Tower, audit logs, capturing all the failures
10.2 Security & Compliance
Pre-Requisite Automation must have strictly read-only access to the MOFSL network shared folder. Write, delete, or move operations are prohibited.
11. Assumptions
All items must be confirmed or corrected by MOFSL during Wave 0 discovery. Any deviation may affect scope, schedule, or cost.
11.1 CBOS, APIs, SMTP
CBOS is up and operational during EDP hours. CBOS downtime is outside CAMS control and will cause EDP Automation failure.
CBOS will expose REST APIs for: EDP process trigger, per-segment checklist trigger, per-sub-process trigger, sub-process status read, and audit log writing. API development is MOFSL IT's responsibility.
CBOS APIs will return structured JSON responses including a machine-readable error code for all failure cases. The error code catalogue must be provided by MOFSL IT.
CBOS APIs will support an idempotency mechanism to prevent duplicate sub-process triggers in retry scenarios. MOFSL IT to confirm and implement.
CBOS API response time SLAs are MOFSL IT's responsibility. Overall EDP cycle duration is dependent on CBOS sub-process execution times.
MOFSL will provide and operate an internal SMTP relay accessible from the CAMS AWS environment for Alerting Automation email dispatch. SMTP connection details (host, port, TLS mode, auth credentials, approved sender address, and DL addresses) must be shared by MOFSL IT before integration testing begins.
MOFSL IT will maintain the distribution lists (DLs) for each alert category and provide DL addresses to Covasant before go-live.
11.2 Infrastructure & Compliance
All hardware and infrastructure (MOFSL AWS tenant, VPC, ECS cluster, IAM roles) are provisioned and maintained by MOFSL. Hardware provisioning is explicitly out of scope for Covasant.
MOFSL will provision network access from the CAMS AWS environment to the on-premises network shared folder (Direct Connect/VPN or equivalent)..
The required file manifest (list of expected files per segment, file naming conventions, folder paths) will be documented and provided by MOFSL IT before Pre-Requisite Automation development begins.
The EDP segment checklists (ordered list of CBOS API calls per segment, step types, retry config) will be documented and provided by MOFSL IT before EDP Automation development begins.
MOFSL will provision dedicated CBOS test user accounts and CAMS service account credentials in Dev+UAT before integration testing begins.
MOFSL will provide UAT test scenarios (covering happy-path, failure, retry, HITL, threshold breach, and all seven segments) to validate all business use cases.

17. Out of Scope
Item
Notes
Automated file download from exchange/depository portals
Files from CDSL, NSDL, NSE, BSE, MCX, NCDEX are still manually downloaded and placed in the network folder. CAMS only monitors for their presence.
CBOS EDP process logic or business rules
All EDP business logic, segment checklist content, and sub-process execution rules are owned by MOFSL IT and CBOS. CAMS Automations execute instructions — they do not define or modify business logic.
Authoring, maintenance, or approval of segment checklists
The EDP Automation consumes pre-approved checklists. Authoring, updating, and approving the checklists is MOFSL's responsibility.
CBOS enhancements beyond EDP API integration
Any changes to CBOS beyond the APIs required for segment triggering, status reading, and audit writing are MOFSL IT's responsibility.
Settlement processing
Post-EDP settlement activities are downstream of EDP Billing and are covered in a separate PRD.
Client-facing communications
The Alerting Automation sends internal operational alerts only. Client-facing communications are out of scope.
Hardware and infrastructure provisioning
Out of scope for Covasant. MOFSL provisions and maintains all AWS infrastructure.
Push notifications and WhatsApp alerting
Planned future channels. Email via SMTP is the only alerting channel in scope for the current release.
MIS or BI reporting dashboards
Analytics dashboards are out of scope. CAMS provides observability via the CAMS Workbench only.
New segment onboarding
Adding new segments beyond the current 7 (Cash, F&O, CD, SLBM, MCX, NCDEX, MTF) is out of scope unless formally agreed via a change request.
