Document Overview
This document provides complete API reference details for the EDP (Exchange Data Processing) Trade Process system. It covers all 26 API steps, from the initial Holiday Master check through to Corporate Action Position Change, including Postman request/response screenshots for key steps.

📌  NOTE: The EQ (Cash) segment is used as the reference example throughout this document. The same sequence applies to all other segments: DR, CUR, SLB, NCDEX, MCX, NSECOM, and MF, unless explicitly stated otherwise.

Base Server URLs
EDP Status API  :  http://10.167.202.234:8087
Main Process API:  http://10.167.202.164:8003
Brokerage API   :  http://10.167.202.164:8003/v1/api/brokerage

High-Level Process Flow

  STEP 1     Check Holiday Master
Before starting any trade process, verify whether today is a holiday for the given segment. If the response returns SKIP, the process must not proceed.


Request Payload
{
    "Segment"     : "EQ",
    "ProcessName" : "BeginFileUpload",
    "UserID"      : "CV0001"
}
Response
{ "Status": "Success", "Data": [{ "MSG": "SKIP" }] }  // Holiday — do NOT proceed
{ "Status": "Success", "Data": [{ "MSG": "TRUE" }] }  // Trading day — proceed

📌  NOTE: If MSG = "SKIP" it is a holiday. Stop here. If MSG = "TRUE" proceed to Step 2.

Postman Screenshot

Step 1 — Holiday Check | Postman

  STEP 2     Get New Trade Process (Create Process ID)
Creates a new trade process for the segment and trade date. Returns a Process ID (Table1) and a full list of upload steps with their statuses (Table2). Pass PROCESSID = 0 to create a new process.


Request Payload
{
    "GROUPNAME" : "EQ",           // Segment Name
    "LOGINID"   : "CV0001",       // User ID
    "TRADEDATE" : "2026-06-29",   // Trade Date (YYYY-MM-DD)
    "PROCESSID" : "0"             // 0 = Create New Process
}
Response
{
    "Status": "Success",
    "Result": {
        "Table1": [{ "PROCESSID": 17658, "ISRUNNABLE": true, "ISAUTOUPLOAD": true }],
        "Table2": [
            { "STEPNO": 1,  "NAME": "Settlement Master NSE Upload", "STATUS": "PENDING", "UPLOADID": 551 },
            { "STEPNO": 2,  "NAME": "Settlement Master BSE Upload", "STATUS": "PENDING", "UPLOADID": 678 },
            { "STEPNO": 3,  "NAME": "BSE Scrip Upload",             "STATUS": "PENDING", "UPLOADID": 81  },
            { "STEPNO": 4,  "NAME": "NSE Scrip Upload",             "STATUS": "PENDING", "UPLOADID": 82  },
            // ... Steps 5-28 all STATUS: PENDING
        ]
    }
}

📌  NOTE: Save the PROCESSID (e.g. 17658) from Table1 — it is required in Steps 6, 8, and all subsequent calls.

  STEP 4     File Upload — Chunk API
Upload the file in chunks. For small files a single chunk (CurrentChunk: 0, TotalChunks: 1) is sufficient. Generate a unique GUID client-side to identify this upload session.


Request Payload
-- Form Data Fields --
file         : <binary file content>
CurrentChunk : 0
TotalChunks  : 1
Guid         : "a1b2c3d4-e5f6-4789-abcd-ef0123456789"
FileName     : "SCRIP_190626.TXT"
Response
{
    "Status"       : "ChunkUploaded",
    "Guid"         : "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "FileName"     : "SCRIP_190626.TXT",
    "currentChunk" : "0",
    "totalChunks"  : "1",
    "fCount"       : "2"
}

📌  NOTE: Save the Guid from the response — it becomes the uploadfoldername field in Step 6.

Step 4 — Chunk File Upload | Postman

  STEP 5     Get Existing Process ID
Retrieves an existing Process ID for a given segment and trade date. Use when resuming an interrupted process.


Request Payload
{
    "TAG"         : "EXISTINGPROCESSID",
    "LOGINID"     : "CV0001",
    "FILTER1"     : "EQ",          // Segment
    "FILTER2"     : "2026-06-29",  // Trade Date (YYYY-MM-DD)
    "extraoption2": "",
    "extraoption3": ""
}
Response
{
    "Status": "Success",
    "Result": [{ "_KEY": 17658, "_DESC": "17658 - CV0001 - Jun 29 2026  2:19PM" }]
}

    "loginid"         : "CV0001",
    "uploadfoldername": "3fa85f64-5717-4562-b3fc-2c963f66afa6",  // GUID from Step 4
    "uploadfilename"  : "SCRIP_190626.TXT",
    "ipaddress"       : "10.167.202.164",
    "file"            : "",
    "paraM1"          : "2026-06-22",  // Trade Date
    "paraM2"          : "", "paraM3": "", "paraM4": "", "paraM5": "",
    "paraM6"          : "", "paraM7": "", "paraM8": "",
    "paraM9"          : "17649",       // Process ID from Step 2
    "chunkFileUpload" : "YES"
}
Response
{ "Status": "Success", "Result": "File entry created successfully." }

  STEP 26     Corporate Action Position Change
This step requires two specific files provided by the Ops Team between 10:00 PM and 11:59 PM. Confirm both are available in the specified location before creating the process.

Required Files (check before proceeding):
  1. _10412_ADJUSTED_POSITIONS.csv
  2. _10412_EXISTING_POSITIONS.csv

📌  NOTE: Both files must be available in the designated location (shared by Ops Team) between 10:00 PM and 11:59 PM before creating the process template.

Step 26a — Create Process (PROCESSID: 0)

{
    "GROUPNAME" : "FOPositionChange",  // Special Segment
    "LOGINID"   : "CV0001",
    "TRADEDATE" : "2026-06-29",
    "PROCESSID" : "0"                  // 0 = Create New
}

Step 26b — File Upload (Repeat Steps 3 → 4 → 6)
Upload both position files using the Chunk Upload flow (Steps 3, 4, 6) with the Upload IDs returned from the FOPositionChange process template.

Step 26c — Execute Process (With Process ID)
{
    "GROUPNAME" : "FOPositionChange",
    "LOGINID"   : "CV0001",
    "TRADEDATE" : "2026-06-29",
    "PROCESSID" : "18756"    // Process ID from Step 26a
}
